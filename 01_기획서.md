### **문서 목적**

본 문서는 **Apple Silicon / macOS 26+ 전용 배터리 충전 제어 앱**의 구현 기준을 정리한다.
작업 주체는 코딩 에이전트이며, 목적은 **AlDente 수준의 핵심 기능을 안정적으로 구현 가능한 구조**를 먼저 확정하는 것이다.

---

## **1. 제품 개요**

### **목표**

이 앱은 macOS에서 배터리 충전 동작을 감시하고, 가능한 환경에서는 **충전 상한 유지 / 재충전 하한 제어 / 일시적 100% 충전 예외 처리**를 제공한다.

### **핵심 가치**

* **사용자 지정 충전 상한 유지**
* **재충전 시작 하한선 지원**
* **일시적 100% 충전**
* **로그인/재부팅 후 정책 복구**
* **메뉴 막대 중심의 빠른 조작**
* **실패 시 안전한 읽기 전용(Read-only) 전환**

### **대상 환경**

* **Apple Silicon 전용**
* **macOS 26 이상**
* **배터리 내장형 MacBook 계열 우선**
* 데스크톱 Mac 및 배터리 미탑재 장비는 비지원 또는 모니터링 전용 처리

---

## **2. 지원 범위 및 호환성 정책**

앱은 설치 가능 여부와 실제 제어 가능 여부를 분리하여 판단한다.

### **지원 원칙**

* 기본 지원 대상: **MacBook Air / MacBook Pro**
* 앱 시작 시 항상 **Capability Probe(기능 가능 여부 점검)** 수행
* 기기/칩/OS 마이너 버전에 따라 기능 지원 범위가 달라질 수 있음

### **비지원/제한 환경 처리**

* 배터리 미탑재 기기: 모니터링 기능 비활성화
* 제어 불가 환경: **Read-only 또는 Monitoring-only 모드**
* 명시적 비호환 환경: 제어 기능 차단 및 사유 표시

## **3. 핵심 기능 요구사항**

### **충전 정책 제어**

### **상태 표시**

* 현재 상태

  * `holdingAtLimit`
  * `waitingForRecharge`
  * `temporaryOverride`
  * `suspended`
  * `errorReadOnly`
* helper 연결 상태
* 최근 정책 적용 결과
* 현재 모드

  * Full Control
  * Read-only
  * Monitoring-only

### **자동화**

* 로그인 시 자동 실행
* 재부팅 후 정책 복구
* 잠자기/깨우기 이후 재동기화

* 정책 변경 직후 즉시 재평가

앱은 아래 기능별 지원 상태를 관리해야 한다.

* 배터리 잔량 조회
* 전원 연결 상태 조회
* 충전 상태 조회

## **5. 안전 정책(Fail-safe)**

이 앱은 배터리 제어 실패 시에도 **무리한 재시도보다 안전한 중단**을 우선한다.

### **원칙**

* 충전 제어 실패 시 즉시 무한 재시도 금지
* helper 통신 실패 시 **Read-only 전환**
* 실제 상태 판독 실패 시 **정책 강제 적용 중단**
* 상태 불일치가 반복되면 자동 비활성화
* 제어 불가 상태를 사용자에게 명확히 알림

### **핵심 기준**

이 앱의 기본 철학은 **“문제가 생기면 제어를 멈추고 관측 모드로 전환”**이다.

## **6. 상태 진실 기준(Source of Truth) 및 재동기화**

앱 내부 상태와 실제 시스템 상태가 다를 수 있으므로 우선순위를 명확히 둔다.

### **우선순위**

1. **실제 시스템에서 읽은 최신 배터리/전원 상태**
2. helper 반환 상태
3. 앱 내부 캐시 상태

### **재동기화 시점**

* 앱 시작 시
* 로그인 직후
* 잠자기 해제 후
* 전원 연결/해제 시
* helper 재시작 후
* 정책 변경 직후
* 상태 불일치 감지 시

### **상태 불일치 처리**

1. 시스템 상태 재조회
2. helper 상태 재조회
3. 내부 상태 재계산
4. 필요 시 Read-only 또는 오류 상태 전환

## **7. 기술 아키텍처**

앱은 **UI / Core / Shared / Helper** 4계층으로 구성한다.

### **AppUI**

* SwiftUI 기반 메뉴 막대 앱
* 상태 표시
* 설정 화면
* 사용자 명령 입력

### **Core**

* 정책 계산
* 상태 머신
* 배터리 상태 해석
* 스케줄링
* capability 판단
* 로그 기록

### **Shared**

* XPC 인터페이스
* 공용 DTO
* 설정/상태 모델

### **Helper**

* 저수준 제어 실행
* self-test
* capability probe
* fallback 처리

### **기술 선택**

* 언어: **Swift**
* UI: **SwiftUI**
* 통신: **XPC**
* 백그라운드 지속성: **Privileged Helper / Launch Daemon**
* 설정 저장: **UserDefaults**
* 로그 저장: **JSON 또는 SQLite**

## **9. 상태 머신 규칙**

### **상태**

* `charging`
* `holdingAtLimit`
* `waitingForRecharge`
* `temporaryOverride`
* `suspended`
* `errorReadOnly`

### **전이 규칙**

* 현재 배터리 >= 상한 → `holdingAtLimit`
* 현재 배터리 <= 재충전 하한 → `charging`
* 예외 충전 활성 → `temporaryOverride`
* 제어 불가 환경 → `suspended`
* helper/XPC 실패 → `errorReadOnly`

상태 머신은 단순하고 테스트 가능하게 유지해야 하며, UI와 분리된 순수 로직으로 구현한다.

## **13. 사용자 보호 UX 규칙**

배터리 제어는 사용자 오해를 줄이는 표현이 중요하다.

### **원칙**

* 상한은 정확히 한 점이 아니라 **허용 구간**일 수 있음
* 실제 충전 중지/재개는 지연될 수 있음
* 정상 제한 유지와 제어 실패를 명확히 구분
* 제어 불가 환경에서는 제어 중처럼 보이게 하면 안 됨

### **예시 문구**

* **80% 제한 유지 중**
* **75% 이하에서 다시 충전 예정**
* **현재 제어 불가: 모니터링만 수행 중**
* **helper 연결 오류로 읽기 전용 모드 전환**

## **최종 결론**

본 앱은 **SwiftUI 메뉴 막대 앱 + Core 정책 엔진 + XPC 기반 권한 분리 Helper + 교체 가능한 저수준 충전 제어 계층** 구조로 설계한다.
핵심은 **기능 구현 자체보다, 호환성 판별, 안전한 fallback, 상태 재동기화, 진단 가능성**을 먼저 확보하는 것이다.
